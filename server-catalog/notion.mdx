The Notion MCP Server gives AI agents within Caylex secure, programmatic access to your Notion workspace, enabling them to read, create, update, and organize content across pages, databases, and views. Agents can search your entire workspace, manage structured data in databases, handle meeting notes, post and retrieve comments, and administer workspace members and teams — all without manual intervention. This server is built on Notion's official hosted MCP infrastructure, meaning it is designed for reliable, production-grade agent workflows covering use cases like project management, knowledge base maintenance, and collaborative note-taking. By connecting this server in Caylex, your AI agents gain the ability to automate end-to-end Notion workflows, from querying database views and duplicating pages to moving content and managing data sources.

<Warning>
**Prerequisites:**
- A Notion account with access to the workspace you want to connect; the integration must be authorized via OAuth to act on behalf of a user.
- Relevant pages and databases must be shared with the Notion integration before agents can read or write to them.
- Workspace admin access may be required for certain operations such as retrieving team or user information via notion-get-teams and notion-get-users.
</Warning>

<Note>
**Important API details:**
- Agents can only access Notion content that has been explicitly shared with the connected integration — pages or databases not shared with the integration will be invisible to the agent.
- Notion's API uses cursor-based pagination for listing and querying resources; agents using tools like notion-query-database-view should handle paginated responses when working with large datasets.
- The Notion API enforces rate limits on a per-integration basis; high-frequency agent workflows that make many rapid requests may encounter throttling.
</Note>

  <Accordion title="notion-query-meeting-notes">
    Query the current user's meeting notes data source.
Applies a filter over meeting note properties. Title keyword searching is done via filter on property "title" (e.g. string_contains). Title keyword matching is case-insensitive; capitalization does not matter. Returns up to 50 rows of matching meeting notes.
Prerequisites:
1. Use the "search" tool to find people IDs if you need to filter by attendees

Query building:
<ul>
<li>Ignore terms semantically related to meeting outputs (e.g. "summaries", "notes", "todos", "action items", "deliverables"). These signal the user wants outcomes from their meetings, not a title filter.</li>
<li>For example, "what are my meeting todos?" means filter meetings and find action items — do NOT add a title filter for "todos".</li>
<li>Only add a title filter when confident the user is targeting a specific meeting title (e.g. "standup", "sprint planning", "1:1 with Alice").</li>
<li>Generic date phrases like "recent meetings", "latest meetings", "meetings this week", or "yesterday's meetings" should be interpreted as date range filters — never as title filters.</li>
<li>If a filter returns no results, simplify to a single term. The system is lexical, so multi-word title filters may not match.</li>
<li>Unless a user explicitly asks about a meeting titled with another user's name, assume they're referring to attendees or creators. Only add a title filter with a person's name as a fallback if attendee filtering returns no results.</li>
</ul>

Default behavior:
<ul>
<li>This tool by default returns meeting notes where the current user is an attendee or creator. There is no need to add a filter for the current user.</li>
</ul>

Filterable properties:
<ul>
<li>"title" (text) — meeting title</li>
<li>"notion://meeting_notes/attendees" (person) — meeting attendees</li>
<li>"created_time" (date) — when the meeting note was created</li>
<li>"created_by" (person) — who created the meeting note</li>
<li>"last_edited_time" (date) — when the meeting note was last edited</li>
<li>"last_edited_by" (person) — who last edited the meeting note</li>
</ul>

Combinator filters use "filters" (not "operands"): \{ "operator": "and" | "or", "filters": [ ... ] \}
Date filtering (recommended default: date_is_within):
<ul>
<li>Prefer "date_is_within" for relative windows like "past N days/weeks/months".</li>
<li>Relative (common): \{ type: "relative", value: "the_past_week" | "the_past_month" | "this_week" \}</li>
<li>Relative (custom): \{ type: "relative", value: "custom", direction: "past" | "future", unit: "day" | "week" | "month" | "year", count: &lt;number&gt; \}</li>
<li>Exact range: \{ type: "exact", value: \{ type: "daterange", start_date: "YYYY-MM-DD", end_date: "YYYY-MM-DD" \} \}</li>
<li>Single-date operators ("date_is", "date_is_before", "date_is_after", "date_is_on_or_before", "date_is_on_or_after"):</li>
<li>Exact: \{ type: "exact", value: \{ type: "date", start_date: "YYYY-MM-DD" \} \}</li>
<li>Relative shortcuts: today | tomorrow | yesterday | one_week_ago | one_week_from_now | one_month_ago | one_month_from_now</li>
</ul>

Title keyword filtering (OR vs AND):
<ul>
<li>Use OR ("operator": "or") when unsure or for broad discovery.</li>
<li>Use AND ("operator": "and") when the user is specific and you want to narrow results.</li>
<li>Break multi-word phrases into individual terms and filter on each term separately.</li>
</ul>

Example 1: Filter meetings from the past week (relative): \{ "filter": \{ "operator": "and", "filters": [ \{ "property": "created_time", "filter": \{ "operator": "date_is_within", "value": \{ "type": "relative", "value": "the_past_week" \} \} \} ] \} \}
Example 2: Filter meetings from the past 3 days (custom relative): \{ "filter": \{ "operator": "and", "filters": [ \{ "property": "created_time", "filter": \{ "operator": "date_is_within", "value": \{ "type": "relative", "value": "custom", "direction": "past", "unit": "day", "count": 3 \} \} \} ] \} \}
Example 3: Filter meetings by exact date range: \{ "filter": \{ "operator": "and", "filters": [ \{ "property": "created_time", "filter": \{ "operator": "date_is_within", "value": \{ "type": "exact", "value": \{ "type": "daterange", "start_date": "2025-01-01", "end_date": "2025-12-31" \} \} \} \} ] \} \}
Example 4: Filter meetings created after a specific date: \{ "filter": \{ "operator": "and", "filters": [ \{ "property": "created_time", "filter": \{ "operator": "date_is_after", "value": \{ "type": "exact", "value": \{ "type": "date", "start_date": "2025-06-01" \} \} \} \} ] \} \}
Example 5: Filter meetings by a specific attendee (use "search" tool first to get user ID): \{ "filter": \{ "operator": "and", "filters": [ \{ "property": "notion://meeting_notes/attendees", "filter": \{ "operator": "person_contains", "value": [ \{ "type": "exact", "value": \{ "table": "notion_user", "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890" \} \} ] \} \} ] \} \}
Example 6: Combine attendees with date range: \{ "filter": \{ "operator": "and", "filters": [ \{ "property": "created_time", "filter": \{ "operator": "date_is_on_or_after", "value": \{ "type": "exact", "value": \{ "type": "date", "start_date": "2025-01-01" \} \} \} \}, \{ "property": "created_time", "filter": \{ "operator": "date_is_on_or_before", "value": \{ "type": "exact", "value": \{ "type": "date", "start_date": "2025-01-31" \} \} \} \}, \{ "property": "notion://meeting_notes/attendees", "filter": \{ "operator": "person_contains", "value": [ \{ "type": "exact", "value": \{ "table": "notion_user", "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890" \} \} ] \} \} ] \} \}
Example 7: Filter meetings by title content: \{ "filter": \{ "operator": "and", "filters": [ \{ "property": "title", "filter": \{ "operator": "string_contains", "value": \{ "type": "exact", "value": "design review" \} \} \} ] \} \}
Example 8: Filter meetings matching any of several title terms (using "or"): \{ "filter": \{ "operator": "or", "filters": [ \{ "property": "title", "filter": \{ "operator": "string_contains", "value": \{ "type": "exact", "value": "standup" \} \} \}, \{ "property": "title", "filter": \{ "operator": "string_contains", "value": \{ "type": "exact", "value": "sync" \} \} \} ] \} \}
  </Accordion>
</AccordionGroup>

<AccordionGroup>
  <Accordion title="notion-create-pages">
    ## Overview
Creates one or more Notion pages, with the specified properties and content.
## Parent
All pages created with a single call to this tool will have the same parent. The parent can be a Notion page ("page_id") or data source ("data_source_id"). If the parent is omitted, the pages are created as standalone, workspace-level private pages, and the person that created them can organize them later as they see fit.
If you have a database URL, ALWAYS pass it to the "fetch" tool first to get the schema and URLs of each data source under the database. You can't use the "database_id" parent type if the database has more than one data source, so you'll need to identify which "data_source_id" to use based on the situation and the results from the fetch tool (data source URLs look like collection://&lt;data_source_id&gt;).
If you know the pages should be created under a data source, do NOT use the database ID or URL under the "page_id" parameter; "page_id" is only for regular, non-database pages.
## Content
Notion page content is a string in Notion-flavored Markdown format.
Don't include the page title at the top of the page's content. Only include it under "properties".
**IMPORTANT**: For the complete Markdown specification, always first fetch the MCP resource at `notion://docs/enhanced-markdown-spec`. Do NOT guess or hallucinate Markdown syntax. This spec is also applicable to other tools like update-page and fetch.
## Properties
Notion page properties are a JSON map of property names to SQLite values.
When creating pages in a database:
<ul>
<li>Use the correct property names from the data source schema shown in the fetch tool results.</li>
<li>Always include a title property. Data sources always have exactly one title property, but it may not be named "title", so, again, rely on the fetched data source schema.</li>
</ul>

For pages outside of a database:
<ul>
<li>The only allowed property is "title",	which is the title of the page in inline markdown format. Always include a "title" property.</li>
</ul>

**IMPORTANT**: Some property types require expanded formats:
<ul>
<li>Date properties: Split into "date:\{property\}:start", "date:\{property\}:end" (optional), and "date:\{property\}:is_datetime" (0 or 1)</li>
<li>Place properties: Split into "place:\{property\}:name", "place:\{property\}:address", "place:\{property\}:latitude", "place:\{property\}:longitude", and "place:\{property\}:google_place_id" (optional)</li>
<li>Number properties: Use JavaScript numbers (not strings)</li>
<li>Checkbox properties: Use "__YES__" for checked, "__NO__" for unchecked</li>
</ul>

**Special property naming**: Properties named "id" or "url" (case insensitive) must be prefixed with "userDefined:" (e.g., "userDefined:URL", "userDefined:id")
## Templates
When creating a page in a database, you can apply a template to pre-populate it with content and property values. Use the "fetch" tool on a database to see available templates in the &lt;templates&gt; section of each data source.
When using a template:
<ul>
<li>Pass the template's ID as "template_id" in the page object.</li>
<li>Do NOT include "content" when using a template, as the template provides it.</li>
<li>You can still set "properties" alongside the template to override template defaults.</li>
<li>Template application is asynchronous. The page is created immediately but starts blank; the template content will appear shortly after.</li>
</ul>

## Icon and Cover
Each page can optionally have an icon and a cover image.
<ul>
<li>"icon": An emoji character (e.g. "🚀"), a custom emoji by name (e.g. ":rocket_ship:"), or an external image URL. Use "none" to remove. Omit to leave unchanged.</li>
<li>"cover": An external image URL. Use "none" to remove. Omit to leave unchanged.</li>
</ul>

### Preserving Child Pages and Databases
When using "replace_content", the operation will check if any child pages or databases would be deleted. If so, it will fail with an error listing the affected items.
To preserve child pages/databases, include them in new_str using `&lt;page url="..."&gt;` or `&lt;database url="..."&gt;` tags. Get the exact URLs from the "fetch" tool output.
**CRITICAL**: To intentionally delete child content: if the call failed with validation and requires `allow_deleting_content` to be true, DO NOT automatically assume the content should be deleted. ALWAYS show the list of pages to be deleted and ask for user confirmation before proceeding.

  <Accordion title="notion-create-database">
    Creates a new Notion database using SQL DDL syntax.
If no title property provided, "Name" is auto-added. Returns Markdown with schema, SQLite definition, and data source ID in &lt;data-source&gt; tag for use with update_data_source and query_data_sources tools.
The schema param accepts a CREATE TABLE statement defining columns.
Type syntax:
<ul>
<li>Simple: TITLE, RICH_TEXT, DATE, PEOPLE, CHECKBOX, URL, EMAIL, PHONE_NUMBER, STATUS, FILES</li>
<li>SELECT('opt':color, ...) / MULTI_SELECT('opt':color, ...)</li>
<li>NUMBER [FORMAT 'dollar'] / FORMULA('expression')</li>
<li>RELATION('data_source_id') — one-way relation</li>
<li>RELATION('data_source_id', DUAL) — two-way relation</li>
<li>RELATION('data_source_id', DUAL 'synced_name') — two-way with synced property name</li>
<li>RELATION('data_source_id', DUAL 'synced_name' 'synced_id') — two-way with synced name and ID (for self-relations)</li>
<li>ROLLUP('rel_prop', 'target_prop', 'function')</li>
<li>UNIQUE_ID [PREFIX 'X'] / CREATED_TIME / LAST_EDITED_TIME</li>
<li>Any column: COMMENT 'description text' Colors: default, gray, brown, orange, yellow, green, blue, purple, pink, red</li>
</ul>

&lt;example description="Minimal"&gt;\{"schema": "CREATE TABLE ("Name" TITLE)"\}&lt;/example&gt;
&lt;example description="Task DB"&gt;\{"title": "Tasks", "schema": "CREATE TABLE ("Task Name" TITLE, "Status" SELECT('To Do':red, 'Done':green), "Due Date" DATE)"\}&lt;/example&gt;
&lt;example description="With parent and options"&gt;\{"parent": \{"page_id": "f336d0bc-b841-465b-8045-024475c079dd"\}, "title": "Projects", "schema": "CREATE TABLE ("Name" TITLE, "Budget" NUMBER FORMAT 'dollar', "Tags" MULTI_SELECT('eng':blue, 'design':pink), "Task ID" UNIQUE_ID PREFIX 'PRJ')"\}&lt;/example&gt;
&lt;example description="Self-relation (two-step: create database first, then use its data source ID with update_data_source to add self-relations)"&gt;\{"title": "Tasks", "schema": "CREATE TABLE ("Name" TITLE, "Parent" RELATION('ds_id', DUAL 'Children' 'children'), "Children" RELATION('ds_id', DUAL 'Parent' 'parent'))"\}&lt;/example&gt;

  <Accordion title="notion-update-data-source">
    Update a Notion data source's schema, title, or attributes using SQL DDL statements. Returns Markdown showing updated structure and schema.
Accepts a data source ID (collection ID from fetch response's &lt;data-source&gt; tag) or a single-source database ID. Multi-source databases require the specific data source ID.
The statements param accepts semicolon-separated DDL statements:
<ul>
<li>ADD COLUMN "Name" &lt;type&gt; - add a new property</li>
<li>DROP COLUMN "Name" - remove a property</li>
<li>RENAME COLUMN "Old" TO "New" - rename a property</li>
<li>ALTER COLUMN "Name" SET &lt;type&gt; - change type/options</li>
</ul>

Same type syntax as create_database. Key types:
<ul>
<li>SELECT('opt':color, ...) / MULTI_SELECT('opt':color, ...)</li>
<li>NUMBER [FORMAT 'dollar'] / FORMULA('expression')</li>
<li>RELATION('ds_id') / RELATION('ds_id', DUAL) / RELATION('ds_id', DUAL 'synced_name' 'synced_id')</li>
<li>ROLLUP('rel_prop', 'target_prop', 'function') / UNIQUE_ID [PREFIX 'X']</li>
<li>Simple: TITLE, RICH_TEXT, DATE, PEOPLE, CHECKBOX, URL, EMAIL, PHONE_NUMBER, STATUS, FILES</li>
</ul>

&lt;example description="Add properties"&gt;\{"data_source_id": "f336d0bc-b841-465b-8045-024475c079dd", "statements": "ADD COLUMN "Priority" SELECT('High':red, 'Medium':yellow, 'Low':green); ADD COLUMN "Due Date" DATE"\}&lt;/example&gt;
&lt;example description="Rename property"&gt;\{"data_source_id": "f336d0bc-b841-465b-8045-024475c079dd", "statements": "RENAME COLUMN "Status" TO "Project Status""\}&lt;/example&gt;
&lt;example description="Remove property"&gt;\{"data_source_id": "f336d0bc-b841-465b-8045-024475c079dd", "statements": "DROP COLUMN "Old Property""\}&lt;/example&gt;
&lt;example description="Add self-relation"&gt;\{"data_source_id": "f336d0bc-b841-465b-8045-024475c079dd", "statements": "ADD COLUMN "Parent" RELATION('f336d0bc-b841-465b-8045-024475c079dd', DUAL 'Children' 'children'); ADD COLUMN "Children" RELATION('f336d0bc-b841-465b-8045-024475c079dd', DUAL 'Parent' 'parent')"\}&lt;/example&gt;

  <Accordion title="notion-create-view">
    Create a new view on a Notion database.
Use "fetch" first to get the database_id and data_source_id (from &lt;data-source&gt; tags in the response).
Supported types: table, board, list, calendar, timeline, gallery, form, chart, map, dashboard.
The optional "configure" param accepts a DSL for filters, sorts, grouping,
and display options. See the notion://docs/view-dsl-spec resource for full
syntax. Key directives:
<ul>
<li>FILTER "Property" = "value" — filter rows</li>
<li>SORT BY "Property" ASC — sort rows</li>
<li>GROUP BY "Property" — group by property (required for board views)</li>
<li>CALENDAR BY "Property" — date property (required for calendar views)</li>
<li>TIMELINE BY "Start" TO "End" — date range (required for timeline views)</li>
<li>MAP BY "Property" — location property (required for map views)</li>
<li>CHART column|bar|line|donut|number — chart type with optional AGGREGATE, COLOR, HEIGHT, SORT, STACK BY, CAPTION</li>
<li>FORM CLOSE|OPEN — close/open form submissions</li>
<li>FORM ANONYMOUS true|false — toggle anonymous submissions</li>
<li>FORM PERMISSIONS none|reader|editor — set submission permissions</li>
<li>SHOW "Prop1", "Prop2" — set visible properties</li>
<li>COVER "Property" — cover image property</li>
</ul>
